# 에이전트 설정 화면 UX 재구성

## 요약

- spec-05에서 추가된 멀티 CLI 에이전트 기능은 유지하고, `src/renderer/features/agent-cli-settings` 화면의 정보 구조와 조작 방식을 다시 설계한다.
- 현재의 상단 요약 카드 + 기본 에이전트 요약 + 긴 설정 카드 목록 구조를 `에이전트 목록 + 선택된 에이전트 상세 인스펙터` 구조로 바꾸고, 반복 설명은 툴팁 또는 on-demand help로 이동한다.

## 배경 / 문제

- `src/renderer/features/agent-cli-settings/AgentCliSettingsPage.tsx` 는 설정 개요, 기본 에이전트 요약, 연결 카드 목록이 수직으로 누적되어 있어 첫 화면에서 정보 밀도가 높고 작업 우선순위가 흐려진다.
- `src/renderer/features/agent-cli-settings/components/AgentCliConnectionCard.tsx` 는 에이전트 설명, PATH 힌트, 상태 요약, 필드가 모두 인라인으로 노출되어 `sdd-ui-disclosure` 기준의 지연 노출 원칙과 맞지 않는다.
- 저장, 기본 지정, 연결 확인 액션이 텍스트 버튼 위주로 배치돼 있고, 저장 가능한 상태와 선택 상태보다 설명 문구가 더 먼저 보인다.
- 저장소에는 이미 `src/renderer/components/IconActionButton.tsx`, `src/renderer/components/TooltipSurface.tsx`, `src/renderer/features/project-bootstrap/project-bootstrap-page/components/InfoSidebar.tsx` 같은 압축형 조작 패턴이 있는데, 에이전트 설정 화면은 이 패턴을 충분히 재사용하지 않는다.

## 목표

- 에이전트 설정 화면을 "지금 무엇을 바꾸는지"가 바로 보이는 선택 중심 UI로 재구성한다.
- 기본 노출 정보는 에이전트 이름, 연결 상태, 기본 여부, 현재 모델/실행 방식 정도로 줄인다.
- 반복/보조 액션은 아이콘 중심으로 바꾸고, 보조 설명은 툴팁 또는 on-demand help로 이동한다.
- 저장 포맷, IPC, 앱 전역 설정 계약은 유지하고 renderer 중심 리팩터링으로 해결한다.

## 비목표

- 멀티 CLI 에이전트 지원 범위 자체를 다시 설계하지 않는다.
- 앱 전역 설정 스키마나 저장 포맷을 바꾸지 않는다.
- CLI 실행, 연결 확인 로직, capability 판단 규칙을 새로 추가하지 않는다.
- 명세 채팅 패널이나 분석 화면의 정보 구조를 이번 범위에서 함께 전면 재설계하지 않는다.

## 사용자 시나리오

- 사용자는 설정 화면에 들어오면 상단 설명을 읽지 않아도 에이전트 목록에서 현재 기본 에이전트와 연결 상태를 바로 확인한다.
- 사용자는 목록에서 `Claude Code` 를 선택하면 오른쪽 또는 하단 인스펙터에서 해당 에이전트 설정만 집중해서 수정한다.
- 사용자는 실행 경로, 인증 방식, 모델을 바꾸고 아이콘 중심 액션으로 저장하거나 연결 확인을 실행한다.
- 사용자는 PATH 탐색 규칙이나 인증 방식 의미가 필요할 때만 툴팁 또는 도움말을 연다.
- 모바일 폭에서는 목록과 상세 패널이 세로로 재배치돼도 같은 흐름으로 작업할 수 있다.

## 기능 요구사항

- `src/renderer/features/agent-cli-settings/AgentCliSettingsPage.tsx` 는 별도 overview 카드와 기본 에이전트 요약 카드를 기본 구조에서 제거하고, 에이전트 선택 영역과 선택된 에이전트 상세 설정 영역을 중심으로 재구성해야 한다.
- 비선택 에이전트는 compact row/card 형태로 이름, 상태, 기본 여부, 핵심 설정 요약만 보여주고 상세 폼은 펼치지 않는다.
- 선택된 에이전트만 상세 설정 폼을 보여주며, 폼 상단에는 `기본 에이전트 지정`, `연결 확인`, `저장`, `새로고침`, `도움말` 같은 반복 액션을 아이콘 중심으로 제공해야 한다.
- 반복/보조 액션은 가능하면 `src/renderer/components/IconActionButton.tsx` 와 `src/renderer/components/TooltipSurface.tsx` 를 재사용한다.
- 다음 문구는 `move to on-demand help` 로 분류한다.
- 에이전트 설명 문장
- PATH 탐색 규칙 설명
- 인증 방식 일반 설명
- 모델/추론 강도 보조 설명
- 다음 문구는 `keep inline` 으로 분류한다.
- 저장 실패 또는 연결 확인 실패 메시지
- 현재 저장 중 또는 연결 확인 중 상태
- 지원하지 않는 capability 때문에 필드가 숨겨지거나 비활성화되는 이유
- 에이전트가 없거나 로딩 실패한 빈 상태의 다음 행동
- 다음 문구는 `remove entirely` 로 분류한다.
- 선택 상태와 카드 메타 요약에서 이미 반복되는 overview 문장
- 기본 에이전트 요약 카드에서 다시 반복되는 상태 문구
- `src/renderer/features/agent-cli-settings/components/AgentCliConnectionCard.tsx` 의 메타 정보는 2열 요약 타일 전체를 유지하기보다, 선택된 에이전트 기준의 핵심 상태만 남기고 나머지는 요약 행 또는 tooltip-trigger로 축약해야 한다.
- `commandMode`, `authMode`, `model`, `modelReasoningEffort` 필드는 capability와 현재 선택 상태를 기준으로 dense inspector 형태로 정렬해야 하며, 사용하지 않는 필드는 숨긴다.
- `기본으로 사용`, `다시 불러오기` 같은 보조 텍스트 버튼은 기본적으로 icon-first control로 전환하고, 접근성을 위해 `aria-label` 과 tooltip을 제공해야 한다.
- 최초 진입 빈 상태나 치명적 오류 복구처럼 텍스트가 없으면 사용자가 막히는 상황에서는 짧은 inline 문구를 허용한다.
- 960px 이하에서는 현재처럼 단순 세로 나열로 무너지지 않도록, 목록 영역과 상세 영역이 순서만 바뀐 단일 흐름으로 재배치되어야 한다.

## 비기능 요구사항

- `renderer` 만으로 해결 가능한 정보 구조/표현 변경을 우선하고, `main`, `preload`, `application` IPC 계약은 불필요하게 바꾸지 않는다.
- 기존 설정 저장/연결 확인 동작은 시각 구조가 바뀌어도 그대로 유지돼야 한다.
- 아이콘 버튼은 마우스 hover 뿐 아니라 키보드 focus 로도 tooltip 에 접근 가능해야 한다.
- 기본 노출 카피는 최소화하되, 접근성 이름(`aria-label`)과 상태 전달은 줄이지 않는다.
- 기존 `IconActionButton` / `TooltipSurface` / 공통 스타일과 충돌하지 않는 범위에서 재사용하고, 설정 전용 스타일은 `src/renderer/styles/settings.css` 와 `src/renderer/styles/responsive.css` 안에서 정리한다.
- 다크 모드와 모바일 폭에서도 현재 상태 배지와 액션 버튼의 구분이 유지돼야 한다.
- 저장 포맷, 스키마 버전, reference-tag 저장 계약은 이번 명세로 변경하지 않는다.

### 예상 변경

- 설정 화면은 상단 overview + summary + 카드 나열 구조에서 선택 중심 인스펙터 구조로 바뀐다.
- `AgentCliConnectionCard.tsx` 는 현재의 긴 설명/힌트/메타 카드에서 요약 정보와 입력 필드를 분리한 dense component 역할로 축소될 가능성이 크다.
- 보조 액션은 텍스트 버튼보다 icon-first control로 바뀌고, tooltip 기반 설명이 추가된다.
- 상태 pill, 저장 시각, capability 차이 표시는 유지하되, 기본 노출 정보 수는 줄어든다.
- `use-agent-cli-settings-workflow.ts` 는 현재 노출하는 상태와 액션을 그대로 활용하고, 새 IPC나 저장 계약 추가는 우선 요구하지 않는다.
- `InfoSidebar.tsx` 의 현재 compact runtime control 패턴은 설정 화면 쪽 재구성의 시각적 기준으로 참고할 수 있다.

### 리스크 / 확인 필요

- 저장 액션까지 완전 icon-only 로 가면 최초 사용자의 발견 가능성이 떨어질 수 있으므로, primary action만은 icon + 강한 affordance 형태가 더 적절한지 확인이 필요하다.
- 에이전트 설명을 모두 tooltip 으로 옮기면 지원 범위 차이를 처음 보는 사용자가 헷갈릴 수 있으므로, capability badge 또는 짧은 상태 문구가 최소 한 줄은 필요한지 검토가 필요하다.
- 현재 `definition.description` 이 한 문장 설명만 제공하므로, tooltip 문안과 inline 상태 문구를 완전히 분리하려면 renderer util 또는 도메인 설명 모델 분리가 필요할 수 있다.
- `AppViewSwitcher` 는 현재 텍스트 탭 형태인데, 이번 명세 범위에 포함할지 여부는 아직 보류한다. 기본안은 설정 본문만 정리하고 전역 뷰 전환은 유지한다.

## 사용 스킬

- `sdd-ui-disclosure`: 이 명세의 핵심이다. visible copy 최소화, tooltip-first 설명 이동, `keep inline / move to on-demand help / remove entirely` 분류에 사용한다.
- `sdd-product-mvp`: Fits current MVP. 설정 화면 UX 정리는 현재 앱 전역 CLI 에이전트 연결 흐름 안에 그대로 들어간다.
- `sdd-implementation-guardrails`: renderer 중심 리팩터링으로 해결하고 IPC/저장 경계를 건드리지 않도록 확인할 때 사용한다.
- `sdd-engineering-conventions`: 설정 feature를 `page + components + styles` 수준으로 정리하고, 공통 UI primitive 재사용 규칙을 지킬 때 사용한다.

## 수용 기준

- [ ] 설정 화면 첫 진입 시 overview 설명 카드 없이도 현재 기본 에이전트와 연결 상태를 바로 이해할 수 있다.
- [ ] 사용자는 한 번에 하나의 에이전트 설정에 집중할 수 있고, 비선택 에이전트는 compact summary 로만 보인다.
- [ ] 에이전트 설명, PATH 안내, 인증 방식 설명 등 반복 설명은 기본 노출에서 빠지고 tooltip 또는 on-demand help 로 이동한다.
- [ ] 저장, 연결 확인, 기본 지정, 새로고침 같은 보조 액션은 icon-first control로 제공되고 `aria-label` 과 tooltip 이 붙는다.
- [ ] 오류, 저장 중 상태, capability 제한 이유, 빈 상태 다음 행동은 inline 으로 유지된다.
- [ ] 기존 설정 저장/불러오기/연결 확인 동작과 선택된 기본 에이전트 저장은 UI 재구성 후에도 동작이 바뀌지 않는다.
- [ ] 960px 이하 폭에서도 목록과 상세 설정 흐름이 유지되고 액션 접근성이 깨지지 않는다.

## 오픈 질문

- 저장 액션을 완전 icon-only 로 둘지, 아이콘 + 짧은 텍스트 primary button 으로 둘지 최종 결정이 필요하다.
- 선택된 에이전트의 capability 차이를 badge 중심으로 보여줄지, tooltip 중심으로만 숨길지 결정이 필요하다.
- 전역 뷰 전환인 `AppViewSwitcher` 까지 같은 icon-first 톤으로 맞출지, 이번 범위에서는 현행 유지할지 확정이 필요하다.
