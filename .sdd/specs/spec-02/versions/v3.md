# 명세 상세 페이지 버전 이력·Markdown 미리보기·본문 복사 개선

## 요약

- 새 명세 생성 직후 템플릿은 번호가 붙은 버전으로 기록하지 않고 작업 초안으로 보관한다. 첫 실질 저장 또는 첫 채팅 반영부터 `v1` 을 만든다.
- 명세 상세 페이지에서 버전 기록 조회, 현재 초안 대비 비교, 이전 버전 적용, 과거 버전 삭제, 편집 중 Markdown 미리보기, 현재 본문 복사를 지원한다.
- 현재 범위 판단: `Fits current MVP`

## 배경 / 문제

- 현재 `src/infrastructure/sdd/fs-project-storage.repository.ts` 의 `createProjectSpec` 는 새 명세 생성 시 곧바로 `latestVersion = 'v1'` 메타를 만들고 `versions/v1.md` 를 저장한다. 사용자가 아직 아무 내용도 바꾸지 않아도 첫 버전이 소모된다.
- 현재 `src/infrastructure/sdd/fs-project-storage.repository.ts` 의 `saveProjectSpec` 는 저장할 때마다 `createNextProjectSpecVersionId` 를 호출해 버전을 증가시킨다. 내용이 실질적으로 바뀌지 않아도 새 버전이 생길 가능성이 있다.
- 현재 `src/infrastructure/sdd/fs-project-spec-documents.ts` 와 `src/domain/project/project-spec-model.ts` 는 `latestVersion` 이 항상 존재한다고 보고 최신 버전 파일을 바로 읽는다. 그래서 버전 없는 초안 상태를 지금 계약만으로는 정직하게 표현하거나 다시 읽기 어렵다.
- 현재 `src/application/project/send-project-session-message.use-case.ts` 는 채팅 응답으로 갱신된 명세도 `saveProjectSpec` 를 통해 저장한다. 따라서 새 명세를 만든 뒤 첫 채팅 반영이 바로 `v2` 가 되는 구조다.
- 현재 상세 편집 UI인 `src/renderer/features/project-bootstrap/project-bootstrap-page/components/specs-workspace/SpecsWorkspaceDocumentView.tsx` 는 읽기 모드의 Markdown 렌더링과 편집 모드의 `textarea` 를 분리하지만 `작성`, `미리보기`, `분할` 전환이 없고, 툴바도 `편집` 또는 `저장/취소` 만 제공한다.
- 같은 파일과 `src/renderer/features/project-bootstrap/project-bootstrap-page/use-project-bootstrap-workbench.workflow.ts` 의 진행 메시지는 저장이 항상 새 버전을 만든다는 가정에 묶여 있다. `변경 없음`, 첫 저장 `v1`, 충돌 상태를 구분하는 UI가 없다.
- 현재 renderer/main/application 경계에는 `readSpecs`, `createSpec`, `saveSpec` 만 있고, 버전 목록 조회, 특정 버전 읽기, 비교, 적용, 삭제용 IPC/API는 없다. 관련 표면은 `src/shared/ipc/project-ipc.ts`, `src/shared/ipc/project-ipc-api.ts`, `src/preload/index.ts`, `src/main/ipc/project-ipc-registration.ts` 에 모여 있다.
- 현재 상세 화면에는 본문 복사 액션이 없어 사용자가 명세 Markdown 을 다른 문맥에 재사용하려면 수동 선택과 복사에 의존해야 한다.
- 실제 저장소에는 `.sdd/specs/spec-01/versions/v1.md`, `.sdd/specs/spec-01/versions/v2.md` 처럼 버전 파일이 누적되지만, UI에서 이 이력을 확인하거나 되돌릴 방법이 없다.

## 목표

- 새 명세를 버전 없는 작업 초안으로 시작하게 바꾼다.
- 첫 실질 변경 시점부터 버전 번호가 시작되도록 규칙을 정리한다.
- 명세 상세 화면 안에서 버전 기록 조회, 비교, 이전 버전 적용, 과거 버전 삭제를 수행할 수 있게 한다.
- 편집 모드에서 Markdown 결과를 즉시 확인할 수 있게 한다.
- 현재 명세 본문을 빠르게 복사해 다른 대화, 문서, 이슈 문맥에 재사용할 수 있게 한다.
- 채팅 반영과 수동 저장이 같은 버전 규칙을 따르도록 맞춘다.

## 비목표

- Git commit 이력이나 외부 형상관리 도구와 연동하지 않는다.
- Markdown 작성 방식을 WYSIWYG 편집기로 교체하지 않는다.
- 오른쪽 채팅 패널을 버전 관리 패널로 바꾸지 않는다.
- patch/diff 승인 화면 전체를 이번 변경에서 새로 설계하지 않는다.
- 명세 전체 삭제 흐름은 1차 범위에서 필수로 포함하지 않고, 우선 과거 버전 삭제와 이전 버전 적용에 집중한다.
- 렌더링된 HTML 또는 PDF 내보내기, 외부 공유 워크플로 전체를 이번 항목에서 새로 설계하지 않는다.

## 사용자 시나리오

1. 사용자가 오른쪽 패널의 `새 명세` 액션으로 명세를 만든다.
2. 상세 화면은 `미저장 초안` 상태로 열리고, 버전 번호 대신 아직 저장되지 않은 작업 초안임을 보여준다.
3. 사용자는 편집 모드에서 `작성`, `미리보기`, `분할` 중 하나를 선택해 Markdown 을 수정하면서 결과를 바로 확인한다.
4. 사용자는 상세 화면의 `본문 복사` 액션으로 현재 명세 Markdown 원문을 즉시 복사한다. 편집 중이면 현재 편집 초안이 복사된다.
5. 사용자가 처음으로 저장하면 `v1` 이 생성되고 버전 기록 패널에 첫 항목이 나타난다.
6. 이후 사용자는 채팅으로 명세를 갱신하거나 직접 수정해 `v2`, `v3` 를 쌓는다.
7. 사용자는 상세 화면의 `버전 기록` 패널에서 특정 버전을 선택해 현재 초안과 비교하고, 필요한 경우 그 버전을 현재 편집 초안으로 적용한다.
8. 사용자는 더 이상 필요하지 않은 과거 버전을 삭제한다. 단, 현재 기준 버전은 직접 삭제하지 않는다.
9. 읽기 전용 프로젝트에서는 사용자가 버전 기록과 Markdown 미리보기, 본문 복사만 사용하고 저장, 적용, 삭제는 비활성화된 상태로 본다.

## 기능 요구사항

1. 새 명세 생성 시 기본 템플릿은 즉시 번호가 붙은 버전으로 기록하지 않아야 한다.
2. 새 명세 생성 직후의 메타 상태는 `latestVersion` nullable 또는 동등한 명시적 draft 상태만 봐도 버전이 아직 없음을 판단할 수 있어야 하며, 목록, 상세, 카드 UI는 이를 `미저장 초안` 또는 확정된 동등 표현으로 보여줘야 한다.
3. 버전이 아직 없는 명세도 앱 재진입 후 다시 열 수 있도록 현재 작업 초안은 영속 저장되어야 한다.
4. 첫 실질 변경 저장 시 `v1` 을 생성해야 한다.
5. 두 번째 이후 저장은 이전에 영속된 현재 본문과 제목이 실제로 달라진 경우에만 다음 버전을 생성해야 한다.
6. 제목 정규화나 줄바꿈 정리만으로 내용이 동일한 경우에는 새 버전을 만들지 말고 `변경 없음` 결과를 반환해야 한다.
7. 채팅으로 갱신되는 명세도 수동 저장과 같은 규칙을 따라야 한다. 새 명세의 첫 채팅 반영은 `v1` 이고, 실질 변화가 없으면 버전 번호를 올리지 않는다.
8. 저장 결과는 최소 `저장됨`, `변경 없음`, `충돌` 을 renderer 가 구분할 수 있는 typed result 로 전달되어야 한다.
9. `src/renderer/features/project-bootstrap/project-bootstrap-page/components/specs-workspace/SpecsWorkspaceDocumentView.tsx` 에 편집 모드용 보기 전환을 추가하고, 최소 `작성`, `미리보기`, `분할` 세 가지 모드를 지원해야 한다.
10. 편집 중 미리보기는 읽기 모드와 같은 Markdown 렌더링 결과를 사용해야 하며 `Mermaid` fenced block 도 동일하게 처리해야 한다.
11. 상세 화면 툴바 또는 동등한 상단 액션 영역에 `본문 복사` 를 제공해야 한다.
12. `본문 복사` 는 현재 명세 문서의 Markdown 원문 전체를 복사해야 하며, 편집 중에는 현재 편집 초안, 읽기 모드에서는 현재 영속된 본문을 기준으로 동작해야 한다.
13. `본문 복사` 는 저장, 버전 생성, revision 증가, IPC 저장 호출을 유발하면 안 된다.
14. `본문 복사` 는 읽기 전용 프로젝트에서도 사용 가능해야 하며, 성공 또는 실패를 일시적이고 사용자 친화적인 피드백으로 보여줘야 한다.
15. 버전 기록은 오른쪽 채팅 패널이 아니라 명세 상세 페이지 내부의 온디맨드 패널 또는 보조 영역에서 열어야 한다.
16. 버전 기록에는 최소 `버전 번호`, `수정 시각`, `요약`, `현재 기준 여부` 가 표시되어야 한다.
17. 사용자는 버전 기록에서 특정 버전을 선택해 현재 초안과 비교할 수 있어야 한다. 비교는 최소한 Markdown line diff 또는 구조화된 변경 요약 중 하나를 제공해야 한다.
18. 사용자는 과거 버전을 현재 편집 초안으로 적용할 수 있어야 한다. 적용 시 기존 버전 파일을 덮어쓰지 않고, 이후 저장에서만 다음 버전이 생성되어야 한다.
19. 과거 버전 삭제는 지원하되, 유일한 저장 버전 또는 현재 기준으로 사용 중인 최신 버전은 삭제 대상에서 제외해야 한다.
20. 읽기 전용 프로젝트에서는 버전 기록 조회, 비교, 미리보기, 본문 복사만 가능하고 적용, 삭제, 저장은 비활성화되어야 한다.
21. 명세 카드 보드와 상세 헤더의 버전 표시는 버전 없음, 버전 있음, 저장 중 충돌 상태를 구분해 보여줘야 한다.
22. 버전 관련 액션은 typed IPC 를 통해서만 renderer 에 노출해야 하며, renderer 가 `.sdd/` 파일 구조를 직접 다루면 안 된다.
23. 본문 복사는 renderer 로컬 클립보드 처리로 우선 구현하고, 저장 계약이나 `.sdd` 구조를 바꾸는 새 IPC 를 추가하지 않아야 한다.

## 비기능 요구사항

- 저장 규칙은 `meta.json` source of truth, `specs/index.json` rebuildable index, old version immutable 원칙을 유지해야 한다.
- 버전 관련 저장 계약이 바뀌면 `schemaVersion`, `revision`, 충돌 검사 규칙을 함께 점검해야 한다.
- 버전 생성, 적용, 삭제는 모두 atomic write 순서를 따라 중간 실패 시 복구 가능한 상태를 유지해야 하며, `meta.json` 갱신은 마지막 단계에 가깝게 유지해야 한다.
- 편집 중 미리보기는 로컬 렌더링으로 동작해야 하며, 키 입력마다 IPC 또는 파일 저장을 발생시키지 않아야 한다.
- 본문 복사는 renderer 로컬 동작으로 처리해야 하며, 클릭마다 `.sdd` 저장이나 세션 로그 append 를 남기지 않아야 한다.
- 상세 화면의 상시 문구는 최소화하고, 버전 기록 설명은 버튼 또는 패널을 열었을 때만 보여줘야 한다.
- 채팅 패널은 계속 채팅 역할만 담당해야 하며, 버전 관리 정보가 오른쪽 패널을 점유하지 않아야 한다.
- 읽기 전용 여부와 충돌 여부는 renderer 임의 추측이 아니라 inspection 및 typed result 기준으로만 표시해야 한다.
- 회귀 검증은 우선순위를 `domain/state transition -> use case -> repository integration -> IPC contract -> renderer component` 순으로 둔다.
- 현재 `tests/` 디렉터리에 테스트 파일이 보이지 않으므로, 이번 작업에서는 최소한 버전 규칙과 저장 결과를 검증하는 새 테스트 추가를 전제로 해야 한다.

### 리스크 / 확인 필요

- 버전 없는 초기 초안을 어떤 파일 표현으로 영속 저장할지 결정이 필요하다. `meta.json` 확장과 별도 초안 파일 중 선택에 따라 읽기 계약과 복구 규칙이 달라진다.
- `src/infrastructure/sdd/fs-project-spec-documents.ts` 가 현재 `latestVersion` 경로를 곧바로 읽는 구조라서, draft 상태 도입 시 read path 회귀가 가장 먼저 발생할 수 있다.
- no-op 저장 판정 기준을 너무 느슨하게 잡으면 불필요한 버전이 남고, 너무 엄격하게 잡으면 의도한 제목 변경이 누락될 수 있다.
- 채팅 반영과 수동 편집이 동시에 일어나면 `revision` 충돌이 자주 보일 수 있어, 충돌 메시지와 재시도 흐름을 명확히 해야 한다.
- 버전 비교를 line diff 로 시작할지, 렌더링 비교와 변경 요약 조합으로 시작할지 결정이 필요하다.
- 버전 기록 패널을 상시 노출하면 중앙 작업 영역이 과밀해질 수 있으므로, 온디맨드 패널 방식과 정보 밀도를 함께 검토해야 한다.
- 클립보드 접근이 실패하거나 제한된 환경에서도 사용자가 왜 복사되지 않았는지 이해할 수 있는 짧은 오류 피드백이 필요하다.
- 현재 `tests/` 디렉터리에 테스트 파일이 보이지 않아, 저장 규칙 변경을 보호할 최소 회귀 테스트를 같이 세우지 않으면 리그레션 위험이 크다.

## 사용 스킬

- 검토 순서: `sdd-product-mvp -> sdd-storage-contracts -> sdd-implementation-guardrails -> sdd-engineering-conventions -> sdd-ui-disclosure`
- `sdd-product-mvp`: 명세 편집 완성도와 상세 작업 흐름을 다루는 변경이므로 `Fits current MVP` 판단을 유지할 때 사용한다.
- `sdd-storage-contracts`: 버전 없는 초기 초안, 첫 저장 `v1`, 버전 적용 및 삭제 규칙, `meta.json` 과 `specs/index.json` 동기화 규칙을 점검할 때 사용한다.
- `sdd-implementation-guardrails`: renderer, application, infrastructure, IPC 경계를 나누고 버전 관리 로직이 renderer 에 새지 않게 유지할 때 사용한다.
- `sdd-engineering-conventions`: no-op 저장 판정, 저장 결과 타입 분리, 파일 분리, 오류 반환, 테스트 우선순위를 정리할 때 사용한다.
- `sdd-ui-disclosure`: 버전 기록, 비교, 삭제 경고, 복사 피드백, 미리보기 전환을 과밀한 상시 설명 없이 온디맨드 방식으로 설계할 때 사용한다.

## 수용 기준

- [ ] `새 명세` 생성 직후에는 버전 칩이 `v1` 로 표시되지 않고, 버전 없는 작업 초안 상태가 보인다.
- [ ] 버전이 아직 없는 명세도 앱을 다시 열었을 때 현재 초안을 그대로 다시 읽을 수 있다.
- [ ] 새 명세의 첫 실질 저장은 `v1` 을 만들고, 내용이 바뀌지 않은 저장은 새 버전을 만들지 않으며 renderer 가 `변경 없음` 을 구분한다.
- [ ] 채팅으로 갱신되는 명세도 동일한 규칙을 따르며, 새 명세의 첫 채팅 반영은 `v1` 이다.
- [ ] 상세 편집 화면에서 `작성`, `미리보기`, `분할` 모드를 전환할 수 있다.
- [ ] 편집 중 미리보기는 읽기 모드와 같은 Markdown 렌더링 결과를 보여주며 `Mermaid` 도 동일하게 동작한다.
- [ ] 상세 화면 툴바에서 현재 Markdown 본문을 복사할 수 있고, 편집 중에는 현재 편집 초안이 복사된다.
- [ ] 본문 복사는 저장이나 버전 증가를 만들지 않으며, 읽기 전용 프로젝트에서도 사용할 수 있다.
- [ ] 명세 상세 화면 안에서 버전 기록을 열어 버전 목록과 메타 정보를 확인할 수 있다.
- [ ] 사용자는 과거 버전을 현재 초안과 비교하고, 현재 편집 초안으로 적용할 수 있다.
- [ ] 사용자는 과거 버전을 삭제할 수 있지만 현재 기준 버전과 유일한 저장 버전은 삭제할 수 없다.
- [ ] 읽기 전용 프로젝트에서는 버전 기록 조회, 비교, 미리보기, 복사만 가능하고 저장, 적용, 삭제는 비활성화된다.
- [ ] 저장 계약 변경이 발생하면 관련 문서와 타입, IPC 계약이 함께 갱신된다.

## 오픈 질문

- 버전 없는 초기 초안 본문을 `meta.json` 확장으로 저장할지, 다른 영속 표현을 둘지 확정이 필요하다.
- 버전 비교의 1차 형태를 Markdown line diff 로 시작할지, 렌더링 비교와 변경 요약 조합으로 시작할지 결정이 필요하다.
- 과거 버전 삭제를 1차 범위에서 최신이 아닌 버전에만 제한할지, 이후에 명세 전체 삭제까지 확장할지 정해야 한다.
- `미저장 초안` 상태의 표시 문구를 `버전 없음`, `초안`, `미저장 초안` 중 무엇으로 통일할지 UI 카피 결정을 남긴다.
