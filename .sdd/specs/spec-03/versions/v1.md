# 완료 명세 아카이브 표시와 파생 명세 연결선 시각화

## 요약

- 완료된 명세를 `archived` 상태로 전환해 목록, 상세, 명세 맵에서 `보관`으로 표시한다.
- 새 명세가 기존 명세의 파생 작업이나 후속 개발이면 구조화된 관계 메타를 저장하고, 명세 맵에서 방향 있는 연결선으로 보여준다.
- 현재 범위 판단: `Fits current MVP`

## 배경 / 문제

- `src/domain/project/project-spec-model.ts` 는 이미 `draft`, `approved`, `archived` 상태를 정의하고, `src/renderer/features/project-bootstrap/project-bootstrap-page/components/specs-workspace/specs-workspace.utils.ts` 는 `archived` 를 `보관`으로 표시할 수 있다. 하지만 현재 저장소/API/UI 흐름에는 상태를 바꾸는 사용자 액션이 없어 실제 기능으로 연결되지 않는다.
- `src/infrastructure/sdd/fs-project-storage.repository.ts`, `src/infrastructure/sdd/fs-project-spec-documents.ts`, `.sdd/specs/*/meta.json` 은 제목, 버전, 초안 Markdown, 요약만 다루며 명세 간 관계를 저장하지 않는다. 앱을 다시 열면 어떤 명세가 어떤 명세에서 이어졌는지 복원할 근거가 없다.
- `src/renderer/features/project-bootstrap/project-bootstrap-page/components/SpecsWorkspace.tsx` 의 명세 맵은 카드 노드만 배치하고 연결선을 렌더링하지 않는다. 반면 `src/renderer/features/project-bootstrap/project-bootstrap-page/components/AnalysisReferenceMap.tsx` 는 분석 참조선 표시 패턴을 이미 가지고 있다.
- `src/application/project/project.ports.ts`, `src/shared/ipc/project-ipc.ts`, `src/shared/ipc/project-ipc-api.ts`, `src/main/ipc/project-ipc-registration.ts`, `src/preload/index.ts` 는 명세 생성/저장/버전 조회 위주 계약만 노출한다. 보관 표시나 관계 편집처럼 버전을 만들지 않는 메타데이터 변경 경로는 없다.
- 현재 저장 계약 문서와 AGENTS 규칙은 `specs/<spec-slug>/meta.json` 을 명세 source of truth 로 둔다. 따라서 아카이브 상태와 명세 연결선도 renderer 임시 상태가 아니라 spec 메타에 영속 저장돼야 한다.

## 목표

- 사용자가 명세를 `draft <-> archived` 범위에서 전환할 수 있게 한다.
- 명세 간 `파생` 또는 `후속 개발` 관계를 spec 메타에 저장하고 재진입 후에도 복원한다.
- 명세 보드에서 보관 상태와 관계선을 함께 보여줘 후속 작업 흐름을 빠르게 파악하게 한다.
- 상세 문서 화면에서도 연결된 명세를 확인하고 바로 이동할 수 있게 한다.
- Markdown 본문 버전 저장과 별개로 메타데이터만 갱신하는 안정적인 경로를 마련한다.

## 비목표

- 이번 변경에서 전체 상태 머신을 `proposed`, `superseded`, `obsolete` 까지 넓히지 않는다.
- 외부 이슈 트래커, Git, 원격 협업 도구와 명세 관계를 동기화하지 않는다.
- 자유 서술 Markdown 을 파싱해 관계선을 자동 추론하지 않는다.
- 아카이브 전용 별도 화면이나 복잡한 그래프 편집기를 추가하지 않는다.
- 이번 1차에서는 `archived` 를 읽기 전용 잠금 상태로 만들지 않는다. 우선 표시와 분류, 관계 가시화에 집중한다.

## 사용자 시나리오

1. 사용자가 구현과 논의가 끝난 명세를 상세 화면에서 `보관` 상태로 전환한다.
2. 명세 카드는 기존 목록과 맵에 남아 있지만 `보관` 배지와 완화된 시각 상태로 보여 다시 찾기 쉽다.
3. 사용자가 새 명세를 작성하면서 기존 명세가 선행 문맥이라면 상세 화면에서 해당 명세를 `파생` 또는 `후속 개발` 관계로 연결한다.
4. 명세 맵은 기존 명세에서 새 명세로 이어지는 방향 있는 연결선을 보여준다.
5. 사용자는 연결선이나 상세 화면의 관계 목록을 통해 이전 명세로 바로 이동해 맥락을 확인한다.
6. 앱을 다시 열어도 `.sdd/specs/*/meta.json` 에 저장된 보관 상태와 관계선이 그대로 복원된다.

## 기능 요구사항

1. 명세 메타는 본문 저장과 별개로 `status` 와 명세 관계를 갱신할 수 있어야 한다.
2. 메타데이터만 바뀌는 `보관`, `보관 해제`, 관계 추가, 관계 제거는 새 Markdown 버전을 만들지 않아야 하며 `latestVersion`, `currentVersion`, `versions/*` 를 불필요하게 변경하지 않아야 한다.
3. 현재 구현 범위에서 사용자에게 노출되는 상태 전이는 `draft <-> archived` 로 제한한다.
4. `archived` 상태의 명세는 목록, 맵, 상세 헤더에서 `보관` 배지를 보여야 한다.
5. `archived` 상태의 명세는 계속 조회 가능해야 하며 새 명세 관계의 대상으로도 선택할 수 있어야 한다.
6. 명세 관계는 `specs/<spec-slug>/meta.json` 안의 구조화 필드로 저장해야 하며, 각 항목은 최소 `targetSpecId`, `type`, `createdAt` 를 가져야 한다.
7. 1차 관계 타입은 `derived-from` 과 `follow-up-to` 두 종류로 제한하고, UI 라벨은 각각 `파생`, `후속 개발` 로 표시한다.
8. 관계 입력은 현재 명세 기준 편집 방식이어야 한다. 예를 들어 현재 명세가 `spec-03` 이고 `spec-01` 을 `derived-from` 으로 연결하면, 맵에서는 `spec-01 -> spec-03` 방향으로 렌더링한다.
9. 동일한 대상 명세에 같은 관계 타입을 중복 저장하면 안 된다.
10. 자기 자신을 관계 대상으로 저장하면 안 된다.
11. 상세 화면에는 연결된 명세 목록과 관계 타입이 보여야 하며, 사용자는 해당 항목을 눌러 대상 명세로 이동할 수 있어야 한다.
12. 명세 맵은 저장된 관계 메타를 바탕으로 카드 사이 연결선을 그려야 하며, 관계 타입 라벨을 함께 보여야 한다.
13. 현재 `SpecsWorkspace.tsx` 가 노드만 렌더링하는 구조를 유지하더라도, 연결선 계산은 저장된 메타를 기준으로만 수행해야 한다.
14. 관계 대상 명세가 삭제되었거나 읽기에서 누락되면 앱이 깨지면 안 되며, 짧은 unresolved 상태를 보여주고 사용자가 관계를 정리할 수 있어야 한다.
15. 관계 변경이나 아카이브 변경은 `revision` 충돌 검사를 따라야 하며, 다른 변경이 먼저 저장된 경우 충돌을 명시적으로 알려야 한다.
16. 채팅 반영, 직접 저장, 버전 적용, 앱 재진입 후 읽기 흐름은 새 메타 필드를 보존해야 한다.
17. `specs/index.json` 이 관계 요약이나 상태를 함께 담는다면 `meta.json` 과 불일치할 때 항상 `meta.json` 을 기준으로 재생성해야 한다.

## 비기능 요구사항

- 저장 계약은 AGENTS 규칙대로 `specs/<spec-slug>/meta.json` 을 source of truth 로 유지하고, `specs/index.json` 은 재생성 가능한 캐시로 유지해야 한다.
- 명세 메타 계약이 바뀌므로 `schemaVersion` 과 이전 버전 fallback 을 명시해야 한다. 기존 `schemaVersion: 2` 메타는 관계 필드가 없어도 읽을 수 있어야 한다.
- 메타 갱신과 인덱스 재생성은 atomic write 흐름을 따라야 하며, renderer 가 `.sdd` 내부 파일 구조를 직접 다루면 안 된다.
- UI 설명 문구는 최소화한다. 관계 타입 설명은 상시 장문 카피 대신 짧은 라벨, 툴팁, 온디맨드 도움말로 처리한다.
- 명세 맵 연결선은 카드 수가 늘어나도 기본 상호작용을 해치지 않아야 하며, 라벨이 과밀하면 hover 또는 focus 기반 보조 표시로 축소할 수 있어야 한다.
- 테스트 우선순위는 domain/state transition 과 repository integration 을 먼저 잡는다. 최소한 `tests/project-spec-model.test.ts` 와 `tests/fs-project-storage.repository.test.ts` 는 새 상태/관계 계약을 보호하도록 확장돼야 한다.

### 예상 변경

- 저장 계층은 기존 `status` 필드를 실제 사용자 액션으로 노출하고, spec meta 에 관계 배열을 추가하는 방향으로 확장된다.
- 읽기 흐름은 기존 versionless draft 지원을 유지하면서 관계 메타가 없는 기존 spec meta 도 읽을 수 있어야 한다.
- renderer workbench 는 본문 저장과 별도로 메타데이터 갱신 액션을 호출할 수 있어야 하며, 충돌 메시지와 성공 메시지를 분리해야 한다.
- 명세 맵은 현재 카드 배치 로직 위에 관계선 계산과 archived 시각 상태를 얹는 방식으로 바뀐다.
- 상세 화면은 장문의 설명 대신 작은 상태 배지, 관계 편집 트리거, 연결 명세 목록을 중심으로 확장된다.
- 테스트는 domain/repository 쪽에서 status 전이, relation 저장, duplicate/self-reference 방지, legacy read fallback 을 우선 보호해야 한다.
- 문서 계약은 `.sdd` 메타 구조가 바뀌므로 storage, design, convention 문서를 함께 동기화해야 한다.

### 리스크 / 확인 필요

- 관계를 현재 명세 기준 `derived-from` / `follow-up-to` 두 개로 시작할지, 1차부터 더 넓은 타입을 둘지 확정이 필요하다.
- 현재 `SpecsWorkspace.tsx` 는 updatedAt 기반 정렬과 자동 배치만 사용한다. 관계선이 추가되면 카드 위치가 자주 바뀌어 가독성이 떨어질 수 있다.
- `archived` 를 1차에서 단순 조직 상태로 둘지, 이후 편집과 채팅 제한까지 연결할지 경계 확정이 필요하다.
- `src/infrastructure/spec-chat/project-spec-chat-codex-prompt.ts` 는 현재 status 와 version 정도만 전달한다. 관계 메타가 채팅 품질에 필요하면 이후 prompt 입력 확장을 검토해야 한다.
- relation target 이 삭제되거나 slug/id 체계가 바뀐 경우 어떤 unresolved 표시 문구로 정리할지 UI 결정을 남긴다.
- `specs/index.json` 에 관계 요약을 복사해 넣을지, map 과 detail 에 필요한 데이터를 매번 meta 에서만 읽을지 캐시 범위 결정이 필요하다.

  - `specs/<spec-slug>/meta.json` source of truth, schemaVersion, revision 충돌, `specs/index.json` 재생성 규칙을 지킨다.
- `sdd-ui-disclosure`
  - 보관 배지, 관계 타입 라벨, 도움말 노출을 과밀하지 않게 설계한다.
- `sdd-implementation-guardrails`
  - renderer, main, preload, application, infrastructure 경계를 나누고 메타 업데이트를 typed IPC 로 유지한다.
- `sdd-engineering-conventions`
  - metadata-only update 로직, 파일 분리, 오류 반환, 테스트 우선순위를 현재 저장소 규칙에 맞춘다.

## 수용 기준

- [ ] 사용자는 기존 명세를 `보관` 상태로 전환하고 다시 `초안` 상태로 되돌릴 수 있다.
- [ ] `보관` 상태는 명세 목록, 맵, 상세 화면에서 명확히 보인다.
- [ ] 아카이브와 관계 변경만으로는 새 Markdown 버전이 생성되지 않는다.
- [ ] 명세 관계는 `meta.json` 에 구조화되어 저장되고 앱 재진입 후 복원된다.
- [ ] 1차 관계 타입 `파생`, `후속 개발` 이 명확한 라벨로 표시된다.
- [ ] 동일 대상 중복 관계와 자기 자신 관계는 저장되지 않는다.
- [ ] 명세 맵에서 관련 명세 사이에 방향 있는 연결선이 보인다.
- [ ] 상세 화면에서 연결된 명세 목록을 보고 해당 명세로 이동할 수 있다.
- [ ] relation target 이 누락돼도 renderer 가 깨지지 않고 정리 가능한 unresolved 상태를 보여준다.
- [ ] revision 충돌 시 사용자는 메타 변경 실패 이유를 알 수 있다.
- [ ] `tests/project-spec-model.test.ts` 와 `tests/fs-project-storage.repository.test.ts` 가 새 계약을 보호한다.
- [ ] 관련 저장 문서와 규칙 문서가 함께 갱신된다.

## 오픈 질문

- 1차 relation cardinality 를 여러 개 허용으로 둘지, 우선 주 연결 명세 1개로 좁힐지 결정이 필요하다.
- `보관 해제` 시 항상 `draft` 로 복귀시키는 현재 단순 규칙으로 충분한지, 이후 `approved` 흐름까지 고려해야 하는지 확인이 필요하다.
- 명세 맵에서 archived spec 을 기본 표시하되 약하게만 보일지, 별도 필터를 함께 둘지 결정이 남아 있다.
