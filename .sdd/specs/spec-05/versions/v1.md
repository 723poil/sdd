# 멀티 CLI 에이전트 연결 및 명세 채팅 에이전트 선택 지원

## 요약

- 현재 Codex CLI만 지원하는 앱 전역 연결 모델을 `Codex CLI`, `Claude Code`, `Gemini CLI`까지 확장한다.
- 설정 화면, 오른쪽 채팅 패널, 분석/참조 태그/명세 채팅 실행 경로가 선택된 에이전트를 일관되게 반영하도록 정리한다.

## 배경 / 문제

- 현재 도메인 설정 모델 `src/domain/app-settings/agent-cli-connection-model.ts` 는 `AGENT_CLI_IDS = ['codex']` 로 고정돼 있다.
- 설정 화면 `src/renderer/features/agent-cli-settings/AgentCliSettingsPage.tsx` 와 `src/renderer/features/agent-cli-settings/components/AgentCliConnectionCard.tsx` 는 제목, 설명, 플레이스홀더, 힌트 문구가 모두 Codex 전용이다.
- 오른쪽 채팅 패널 `src/renderer/features/project-bootstrap/project-bootstrap-page/components/InfoSidebar.tsx` 는 상단 브랜드가 `CODEX` 로 고정돼 있고, 현재 런타임 설정도 Codex 모델/추론 강도만 전제한다.
- 워크벤치 상태와 API `src/renderer/features/project-bootstrap/project-bootstrap-page/project-bootstrap-workbench.api.ts`, `src/renderer/features/project-bootstrap/project-bootstrap-page/use-project-bootstrap-workbench.workflow.ts` 도 `readCodexConnectionSettings`, `codexConnectionSettings` 같은 이름으로 Codex 전용 흐름을 갖고 있다.
- 분석, 자동 참조 태그, 명세 채팅 실행 어댑터는 `src/infrastructure/analysis/node-project-analyzer-codex-settings.ts`, `src/infrastructure/reference-tags/node-project-reference-tag-generator.adapter.ts`, `src/infrastructure/spec-chat/node-project-spec-chat.adapter.ts` 처럼 Codex 전용 런타임 해석과 실행 파일에 묶여 있다.
- 결과적으로 사용자는 다른 CLI 에이전트를 연결해도 앱에서 선택하거나 실행 경로에 반영할 수 없다.

## 목표

- 앱 전역 설정에서 `Codex CLI`, `Claude Code`, `Gemini CLI` 를 각각 연결 확인하고 저장할 수 있다.
- 사용자는 현재 작업에 사용할 에이전트를 명확히 선택할 수 있고, 채팅 패널과 상태 문구는 선택된 에이전트 기준으로 표시된다.
- 명세 채팅 실행 인자와 분석 계열 실행 경로가 더 이상 Codex 하드코딩에 의존하지 않고, 선택된 에이전트의 런타임 설정을 사용한다.
- 기존 `exec` 중심 구조, `main process` subprocess 실행, typed IPC, 앱 측 `.sdd` 저장 책임은 유지한다.
- 기존 Codex 사용자 설정은 깨지지 않고 그대로 마이그레이션된다.

## 비목표

- `app-server` 또는 `mcp-server` 기반 연결로 이번 범위를 확장하지 않는다.
- 여러 에이전트를 한 turn 에 동시에 실행하거나 결과를 병합하는 멀티 에이전트 오케스트레이션은 다루지 않는다.
- 코드 자동 수정, patch 적용, 테스트 자동 실행 범위까지 이번 명세에서 확장하지 않는다.
- 각 CLI의 고유 고급 기능을 앱 UI에 전부 노출하는 것은 목표가 아니다.

## 사용자 시나리오

- 사용자는 설정 화면에서 `Claude Code` 실행 경로와 인증 방식을 저장하고 연결 확인을 통과한다.
- 사용자는 프로젝트의 명세 페이지 오른쪽 채팅 패널에서 현재 에이전트를 `Gemini CLI` 로 바꾼 뒤 질문을 보내고, 응답과 함께 현재 명세 초안이 갱신된다.
- 사용자는 앱 전역 기본 에이전트를 `Codex CLI` 에서 `Claude Code` 로 바꾸고, 이후 전체 분석과 자동 참조 태그 생성이 새 기본 에이전트를 사용하도록 기대한다.
- 기존 Codex 사용자도 앱을 업데이트한 뒤 추가 설정 파일 수동 수정 없이 그대로 Codex 연결과 기존 모델 설정을 이어서 사용한다.

## 기능 요구사항

- 도메인 설정 모델은 `codex` 단일 값 대신 최소 `codex`, `claude-code`, `gemini-cli` 를 식별할 수 있어야 한다.
- 각 에이전트 정의는 최소 `displayName`, `description`, `defaultExecutableName`, `connectionCheckArgs`, `supportedAuthModes`, `capabilities` 를 가져야 한다.
- 앱 전역 설정 저장소는 여러 에이전트 연결 레코드를 함께 저장하고, 현재 기본 에이전트 식별자도 함께 저장해야 한다.
- 기존 Codex 전용 설정 파일은 새 구조로 읽을 때 자동 보정되며, 기존 `model`, `modelReasoningEffort`, 실행 경로를 유지해야 한다.
- 설정 화면은 Codex 전용 제목 `Codex 연결` 대신 에이전트 중립 표현을 사용하고, 에이전트별 카드 또는 동일 밀도의 목록으로 각 연결 상태를 보여줘야 한다.
- 설정 화면의 실행 경로 힌트, 인증 방식 설명, 모델 선택 문구는 에이전트 정의를 기반으로 바뀌어야 하며, 지원하지 않는 입력은 숨기거나 비활성화해야 한다.
- 오른쪽 채팅 패널은 고정 브랜드 `CODEX` 대신 현재 선택된 에이전트 표시를 사용해야 한다.
- 오른쪽 채팅 패널은 메시지 전송 전에 현재 에이전트를 바꿀 수 있어야 하며, 선택 가능한 모델/추론 강도도 그 에이전트 정의에 맞게 바뀌어야 한다.
- `src/shared/ipc/project-ipc.ts` 의 세션 메시지 전송 입력은 `agentId` 를 포함해야 하며, `src/application/project/send-project-session-message.use-case.ts` 와 이후 실행 계층까지 전달돼야 한다.
- 명세 채팅 실행 어댑터는 선택된 `agentId` 기준으로 올바른 실행기와 출력 파서를 사용해야 한다.
- 전체 분석과 자동 참조 태그 생성도 선택된 에이전트를 사용할 수 있어야 하며, 기능별로 지원하지 않는 에이전트가 있다면 실행 전에 짧고 명확한 이유를 보여줘야 한다.
- 런타임 해석 계층은 `resolveCodexRuntimeSettings` 같은 Codex 고정 함수 대신 공통 에이전트 런타임 해석 계약으로 이동해야 한다.
- 상태 메시지와 오류 메시지는 항상 선택된 에이전트 이름을 포함해 사용자가 어떤 CLI에서 실패했는지 알 수 있어야 한다.

## 비기능 요구사항

- `renderer` 는 CLI 실행 파일 탐색이나 subprocess 실행을 직접 수행하지 않고, 현재처럼 `main process + typed IPC` 경계를 유지해야 한다.
- 앱 전역 설정 JSON 은 `schemaVersion` 기반으로 마이그레이션 가능해야 하고, 기존 사용자 설정을 유실시키지 않아야 한다.
- 사용자 화면의 기본 문구는 최소화하고, 에이전트별 설명이 길어질 경우 툴팁 또는 on-demand 도움말로 미뤄야 한다.
- 프로젝트가 읽기 전용일 때의 기존 정책은 유지돼야 하며, 에이전트가 바뀌어도 분석 임시 표시와 저장 실패 처리 규칙은 동일해야 한다.
- 기능별 capability 차이로 특정 제어가 비활성화될 때는 내부 구현 용어 대신 사용자 행동 기준의 짧은 이유만 보여줘야 한다.
- 테스트 우선순위는 도메인 설정 모델, 앱 설정 저장소 마이그레이션, IPC 계약, 에이전트 런타임 선택, 명세 채팅 전달 인자 순으로 둔다.

### 예상 변경

- 에이전트 정의가 Codex 단일 상수에서 다중 정의 레지스트리로 바뀐다.
- 앱 전역 설정 포맷에 기본 에이전트와 capability-aware 연결 정보가 추가된다.
- 설정 화면 카피와 카드 렌더링이 Codex 전용에서 에이전트 공통 UI로 바뀐다.
- 워크벤치 상태에서 `codexConnectionSettings` 중심 이름과 흐름이 `selectedAgentConnection` 또는 동등한 중립 개념으로 바뀐다.
- 명세 채팅 IPC 입력에 `agentId` 가 추가되고, 세션 메시지 실행 경로가 선택된 에이전트 기준으로 분기된다.
- 분석, 자동 참조 태그, 명세 채팅 실행 계층은 공통 런타임 해석기와 에이전트별 실행 어댑터 구조로 정리될 가능성이 크다.
- 문서와 참조 태그도 `Codex 전용` 표현을 `CLI 에이전트` 중심 표현으로 갱신해야 한다.

### 리스크 / 확인 필요

- `Claude Code` 와 `Gemini CLI` 의 실제 모델 지정 방식, 추론 강도 지원 여부, 구조화 출력 강제 방식이 Codex와 동일하지 않을 수 있다.
- 첨부 파일, JSON 출력, 장시간 실행 관측 방식이 에이전트마다 다르면 capability 매트릭스가 필요하다.
- `~/.codex/config.toml` 기반 기본값 읽기처럼 Codex 전용 fallback 이 이미 존재하므로, 다른 에이전트에 대한 기본값 소스는 별도 정책이 필요하다.
- 기존 Codex 전용 파일명을 즉시 일반화할지, 내부 구현 이름은 유지하고 계약만 일반화할지 결정이 필요하다.
- 첫 릴리스에서 분석, 자동 참조 태그, 명세 채팅을 모두 3개 에이전트에 대해 같은 수준으로 열면 검증 범위가 빠르게 커질 수 있다.

## 사용 스킬

- `sdd-product-mvp`: 현재 요청은 기존 CLI 설정과 명세 채팅 흐름의 확장이므로 현재 MVP 범위에 포함되는지 확인할 때 사용한다.
- `sdd-codex-cli`: 현재 `exec` 중심 연결을 유지한 채 다중 CLI 실행 계약으로 일반화할 때 사용한다.
- `sdd-ui-disclosure`: 설정 화면과 오른쪽 채팅 패널의 문구를 Codex 고정 표현에서 에이전트 중립 표현으로 정리할 때 사용한다.
- `sdd-engineering-conventions`: 도메인, IPC, renderer, infrastructure 경계를 지키면서 파일 책임을 나눌 때 사용한다.
- `sdd-implementation-guardrails`: 실제 구현에서 `main / preload / renderer / application / domain / infrastructure` 경계를 유지할 때 사용한다.
- `sdd-agent-xml-prompt`: 분석/채팅 프롬프트와 구조화 출력 계약을 에이전트별로 분기하거나 공통화할 때 사용한다.

## 수용 기준

- [ ] 설정 화면에서 `Codex CLI`, `Claude Code`, `Gemini CLI` 연결 정보를 각각 저장하고 연결 확인할 수 있다.
- [ ] 기존 Codex 사용자 설정은 수동 마이그레이션 없이 그대로 읽히고 기존 Codex 실행도 유지된다.
- [ ] 오른쪽 채팅 패널은 더 이상 `CODEX` 고정 표기를 쓰지 않고 현재 선택된 에이전트를 명확히 보여준다.
- [ ] 사용자는 채팅 전송 전에 현재 에이전트를 바꿀 수 있고, 모델/추론 강도 입력은 해당 에이전트 capability 에 맞게 조정된다.
- [ ] 세션 메시지 전송, 전체 분석, 자동 참조 태그 생성은 선택된 `agentId` 를 실행 계층까지 전달한다.
- [ ] 지원하지 않는 기능 조합은 실행 전에 짧은 이유와 함께 비활성화되거나 차단된다.
- [ ] 오류와 진행 상태 문구는 실패한 에이전트 이름을 포함해 사용자가 즉시 원인을 구분할 수 있다.
- [ ] 관련 설계 문서와 참조 태그가 Codex 전용 표현에서 다중 CLI 에이전트 표현으로 함께 갱신된다.

## 오픈 질문

- 첫 릴리스 범위를 `명세 채팅 우선` 으로 자를지, `분석/자동 태그/명세 채팅` 3개 흐름을 한 번에 다 열지 확정이 필요하다.
