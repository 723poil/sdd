# 명세 채팅 이미지/파일 첨부 및 드래그 앤 드롭 지원

## 요약

- `Fits current MVP`: 기존 명세 채팅 입력을 확장해 스크린샷 이미지와 로컬 파일을 `붙여넣기` 또는 `드래그 앤 드롭`으로 함께 전송할 수 있게 한다.
- 오른쪽 채팅 패널에서 첨부를 전송 전에 미리 확인하고 제거한 뒤 보낼 수 있어야 하며, assistant 응답과 함께 현재 명세 초안 갱신 흐름을 유지한다.

## 배경 / 문제

- 현재 오른쪽 채팅 패널은 `src/renderer/features/project-bootstrap/project-bootstrap-page/components/InfoSidebar.tsx`의 `textarea` 기반 입력만 제공하고, 붙여넣기나 drop 처리 핸들러는 없다.
- 전송 흐름 `src/renderer/features/project-bootstrap/project-bootstrap-page/use-project-bootstrap-workbench.workflow.ts`는 `draftMessage.trim().length === 0`이면 바로 종료하므로 첨부만 있는 메시지를 보낼 수 없다.
- IPC 계약 `src/shared/ipc/project-ipc.ts`의 `SendProjectSessionMessageInput`과 도메인 계약 `src/domain/project/project-session-model.ts`의 `ProjectSessionMessage`도 현재는 `text`만 다룬다.
- 세션 저장소 `src/infrastructure/sdd/fs-project-session.repository.ts`와 `src/infrastructure/sdd/fs-project-session-documents.ts`는 메시지 JSONL append와 세션 preview 갱신을 텍스트 기준으로만 처리한다.
- `src/infrastructure/spec-chat/node-project-spec-chat.adapter.ts`와 `src/infrastructure/spec-chat/node-project-spec-chat-codex-execution.ts`는 XML prompt 문자열 하나를 `codex exec` 표준 입력으로 넘기는 구조라, 첨부를 별도 입력으로 전달하는 경로가 아직 없다.
- 사용자는 캡처 이미지나 관련 파일을 채팅에 바로 끌어놓거나 붙여 넣어 맥락을 함께 보내고 싶지만, 지금은 텍스트로 다시 설명해야 해서 명세 협업 품질과 속도가 떨어진다.

## 목표

- 명세 채팅에서 클립보드 이미지, 클립보드 파일, Finder 등에서 끌어놓은 로컬 파일을 첨부해 전송할 수 있다.
- 사용자는 전송 전에 첨부 목록과 썸네일을 확인하고 개별 제거할 수 있다.
- 메시지 전송 시 첨부 정보가 세션 로그와 Codex 입력에 반영돼 assistant reply와 spec draft 갱신에 함께 사용된다.
- 텍스트만 보내는 기존 흐름은 그대로 유지한다.

## 비목표

- 자동 OCR, 이미지 주석 편집, PDF 페이지 분할, Office 문서 변환.
- 분석 workspace나 참조 태그 화면으로의 범용 파일 업로드.
- 대용량 바이너리, 폴더 업로드, 클라우드 동기화, 외부 스토리지 연동.
- 채팅 외의 patch/diff 실행 단계 확장.
- 이번 1차 범위에서 별도 업로드 관리자나 첨부 라이브러리 도입.

## 사용자 시나리오

1. 사용자는 프로젝트를 열고 `명세` 페이지에서 작업 중인 명세를 선택한다.
2. 오른쪽 `채팅` 입력창에 스크린샷을 붙여넣거나, 로컬 파일을 붙여넣거나, Finder에서 파일을 끌어놓는다.
3. 입력창 근처에 첨부 칩 또는 썸네일이 표시되고, 사용자는 필요 없는 첨부를 제거한다.
4. 사용자는 설명 텍스트를 추가하거나 텍스트 없이 첨부만 전송한다.
5. 메시지가 전송되면 채팅 타임라인에 텍스트와 첨부 요약이 함께 표시되고, 이미지는 작은 미리보기를 가진다.
6. assistant는 첨부 맥락을 반영한 답변을 반환하고, 현재 명세 초안도 함께 갱신된다.
7. 지원하지 않는 형식 또는 용량 초과면 전송 전에 이유가 짧게 표시되고, 이미 수집된 다른 첨부와 텍스트 입력은 유지된다.

## 기능 요구사항

- 오른쪽 채팅 composer는 `paste`, `dragenter/dragover/dragleave/drop` 이벤트를 처리해 이미지와 파일 첨부를 수집할 수 있어야 한다.
- 붙여넣기와 드래그 앤 드롭은 같은 첨부 draft 상태 모델을 사용해야 하며, 전송 전까지는 renderer 상태에서만 유지한다.
- 1차 지원 대상은 스크린샷 이미지(`png`, `jpg`/`jpeg`, `webp`)와 로컬 텍스트/코드 계열 파일(`txt`, `md`, `json`, `yaml`/`yml`, `toml`, `ts`, `tsx`, `js`, `jsx`, `css`, `html`, `java`, `kt`, `php`, `py`)로 제한한다.
- 메시지는 `텍스트만`, `텍스트+첨부`, `첨부만` 세 경우를 모두 허용한다.
- 첨부는 한 메시지에 여러 개를 둘 수 있고, 전송 전 개별 삭제가 가능해야 한다.
- 드래그 중인 파일이 composer 위에 올라오면 짧은 시각 상태 변화로 drop 가능 영역을 보여주되, 큰 설명 문구를 상시 노출하지 않는다.
- 채팅 타임라인은 각 메시지의 첨부 개수와 이름을 보여주고, 이미지는 작은 미리보기를 제공한다.
- 전송 시점에 main/application/infrastructure 경로가 첨부를 프로젝트 내부 `.sdd/sessions/<session-id>/attachments/...` 아래로 사본 저장하고, 메시지와 연결된 manifest를 생성해야 한다.
- 이미지 첨부는 원본 포맷과 확장자를 최대한 유지한 사본 저장을 기본으로 하고, 표시용 미리보기는 저장된 경로 기준으로 재구성 가능해야 한다.
- 파일 첨부는 원본 파일명을 보존하는 메타데이터와 앱이 참조할 실제 저장 경로, MIME 또는 판별된 kind, 바이트 크기를 남겨야 한다.
- 세션 메시지 저장 계약은 첨부 manifest를 포함하도록 확장해야 하며, 기존 첨부 없는 세션 로그도 계속 읽을 수 있어야 한다.
- `SendProjectSessionMessageInput`, renderer IPC facade, main IPC handler, use case, session repository, spec-chat adapter가 첨부 배열을 함께 전달해야 한다.
- `src/infrastructure/spec-chat/project-spec-chat-codex-prompt.ts`는 최근 메시지 텍스트뿐 아니라 현재 turn의 첨부 정보와 저장 경로를 명시적으로 prompt 입력에 포함해야 한다.
- Codex 응답이 실패해도 사용자 메시지와 첨부 저장은 보존해야 한다.
- 텍스트 전용 전송 UX와 단축 흐름은 회귀 없이 유지해야 한다.
- 지원하지 않는 형식, 개수 초과, 크기 초과, 읽을 수 없는 파일은 전송 전에 개별 항목 단위로 걸러야 하며, 허용된 다른 첨부는 유지해야 한다.
- 현재 저장소 기준으로 채팅 composer에는 drop 처리와 파일 선택 UI가 없으므로, 이번 범위는 `붙여넣기 + 드래그 앤 드롭`을 우선 명시하고 별도 파일 선택 버튼은 선택 사항으로 둔다.

## 비기능 요구사항

- 기본 visible copy는 최소화하고, 형식 또는 용량 설명은 툴팁이나 짧은 오류 문구로만 노출한다.
- renderer는 파일 시스템에 직접 접근하지 않고 typed IPC를 통해 첨부 저장과 전송을 요청해야 한다.
- Codex 실행은 계속 `main process`의 `codex exec` 경로를 사용한다.
- 첨부 저장 관련 JSON 계약은 `schemaVersion`을 가져야 하고, mutable metadata는 기존 `revision` 또는 `updatedAt` 규칙과 충돌하지 않아야 한다.
- 세션 메시지 로그는 append-only 특성을 유지해야 하며, 첨부 추가 때문에 기존 메시지를 재작성하지 않는다.
- 프로젝트가 읽기 전용이면 현재와 동일하게 채팅 자체를 허용하지 않으며, 전송 전 미리보기만으로 저장 성공처럼 보이게 해선 안 된다.
- 초기 제한값 초안은 메시지당 최대 5개, 파일당 최대 20MB, 총합 최대 30MB로 둔다.
- 지원하지 않는 형식 또는 크기 오류는 전송 전에 판별하고, 실패 이유는 사용자 행동 기준으로 짧게 표시한다.
- 드래그 앤 드롭 hover 상태, 첨부 칩, 썸네일은 데스크톱과 좁은 폭 레이아웃 모두에서 composer 기본 동작을 가리지 않아야 한다.
- `codex exec` 입력이 현재 표준 입력 XML prompt 중심이므로, 첨부를 prompt 메타데이터와 저장 경로로 우선 전달하는 설계를 기본안으로 두고 실제 이미지 해석 품질은 검증이 필요하다.

### 예상 변경

- renderer는 `textarea` 단독 입력에서 `텍스트 + 첨부 draft 목록 + drop 상태`를 가진 composer로 확장된다.
- 현재 workbench에는 최근 프로젝트 재정렬용 drag 상태만 존재하므로, 채팅 composer 전용 drag/drop 상태는 별도 분리해야 한다.
- `use-project-bootstrap-workbench.workflow.ts`의 현재 `trimmedDraftMessage.length === 0` 가드가 첨부-only 전송을 막고 있으므로, 전송 가능 조건을 `텍스트 또는 첨부 존재` 기준으로 바꿔야 한다.
- IPC 입력과 출력 계약은 메시지 본문 외에 첨부 manifest 또는 draft 입력 배열을 주고받도록 바뀐다.
- session message schema가 확장되며, 저장소는 첨부 파일 저장과 `messages.jsonl` append, 세션 preview 갱신을 함께 처리해야 한다.
- `project-session-model.ts`의 메시지 schemaVersion 상승과 legacy reader 호환이 필요하다.
- spec chat prompt는 현재 대화 이력 외에 현재 turn 첨부 path, kind, 파일명, 요약 텍스트를 Codex가 읽을 수 있는 구조로 전달해야 한다.
- `node-project-spec-chat-codex-execution.ts`는 여전히 `codex exec` stdin prompt 구조를 유지하므로, 1차 구현은 별도 transport 추가보다 prompt 계약 확장 쪽 영향이 크다.
- 문서 동기화 대상은 `docs/codex-spec-workflow/04-storage-format.md`, `docs/codex-spec-workflow/08-codex-cli-integration.md`가 우선이며, 저장 계약이나 UI 규칙 표현이 바뀌면 `docs/codex-spec-workflow/06-design-patterns.md`, `docs/codex-spec-workflow/07-engineering-conventions.md`도 같이 확인해야 한다.

### 리스크 / 확인 필요

- `codex exec`가 이미지와 파일 첨부를 어떤 형식으로 가장 안정적으로 읽는지 현재 코드만으로는 확정되지 않는다. 현재 adapter는 prompt 문자열만 전달하므로 path 메타데이터만으로 충분한지 검증이 필요하다.
- macOS clipboard의 스크린샷 붙여넣기와 Finder drag-and-drop 파일 입력을 하나의 UX로 묶을 때, 브라우저 이벤트와 Electron 환경 차이를 확인해야 한다.
- 이미지와 파일을 전송 직전에 `.sdd/`로 복사 저장할 때, 대용량 파일 처리 시간과 취소 시 정리 규칙을 정해야 한다.
- 현재 `tests/` 아래에는 session/message 계약 테스트가 없고 `rg -n "session|message" tests` 결과도 없다. domain, use case, repository, IPC 순서로 신규 테스트가 필요하다.

## 사용 스킬

- `sdd-product-mvp`: 이 기능이 현재 MVP의 명세 채팅 입력 확장인지 확인한다.
- `sdd-implementation-guardrails`: renderer, preload, main, application, infrastructure 경계를 지키면서 첨부 흐름을 분리한다.
- `sdd-storage-contracts`: `.sdd/sessions/*` 첨부 저장 위치, 메시지 `schemaVersion`, append-only 규칙을 설계한다.
- `sdd-ui-disclosure`: 오른쪽 채팅 패널의 첨부 도움말, drop 상태, 오류 문구를 과밀하지 않게 조정한다.
- `sdd-agent-xml-prompt`: `src/infrastructure/spec-chat/project-spec-chat-codex-prompt.ts`의 XML 입력 계약에 첨부 컨텍스트를 추가할 때 사용한다.
- `sdd-codex-cli`: `codex exec` 기반 첨부 전달 방식과 main process 실행 제약을 점검한다.
- `sdd-engineering-conventions`: 타입 확장, 파일 분리, 테스트 우선순위를 점검한다.

## 수용 기준

- [ ] 명세 페이지의 오른쪽 채팅 패널에서 스크린샷 이미지 붙여넣기 후 첨부 목록이 보인다.
- [ ] 지원 파일을 붙여넣거나 drag-and-drop 했을 때 전송 전 제거가 가능하다.
- [ ] 드래그 중인 파일을 composer 위로 올리면 drop 가능 상태가 보인다.
- [ ] 텍스트 없이 첨부만으로도 메시지 전송이 가능하다.
- [ ] 전송된 사용자 메시지는 세션 타임라인에 첨부 요약과 함께 보인다.
- [ ] 세션 저장소가 첨부 manifest를 포함한 메시지를 재로드할 수 있다.
- [ ] assistant 응답 생성 시 첨부 맥락이 prompt 입력에 반영되고, 기존처럼 spec draft, title, summary, reply가 반환된다.
- [ ] Codex 응답 실패 시에도 사용자 메시지와 첨부 저장은 남고, 실패 이유가 짧게 표시된다.
