# Product Overview

## 목적

이 도구의 목적은 두 가지다.

1. 기존 프로젝트를 읽고 구조를 분석해 명세 기반 작업이 가능하도록 정리한다.
2. 명세서별 채팅으로 명세를 추가/수정하고, 수정된 명세를 기준으로 이후 작업을 진행할 수 있게 한다.

## 사용자 시나리오

현재 가정하는 사용자는 개발자 본인이다.

- 개발자는 자신의 노트북에서 프로그램을 실행한다.
- 실행 후 화면에서 로컬 프로젝트 폴더를 선택한다.
- 앱이 그 프로젝트를 읽고 구조를 분석한다.
- 분석 결과를 기반으로 명세서 초안을 만든다.
- 개발자는 명세서를 읽고 채팅으로 보완하거나 수정한다.
- 확정된 명세는 이후 Codex 작업 준비의 기준이 된다.

즉, 이 제품은 서버형 서비스가 아니라 "개발자 개인용 로컬 작업 도구"다.

## 구현 가능성

두 기능 모두 구현 가능하다.

### 기능 A

프로젝트 분석 후 명세 기반 작업이 가능하도록 정리하는 기능은 아래 조합으로 구현한다.

- 정적 분석
  - 파일 트리
  - 언어/프레임워크 감지
  - import/dependency 관계
  - 설정 파일 추출
- 규칙 기반 요약
  - 엔트리포인트 후보
  - 디렉터리 역할 분류
  - 테스트/빌드/배포 구조 추정
- LLM 기반 요약
  - 모듈 책임 설명
  - 도메인 개념 정리
  - 변경 영향 요약

핵심 산출물은 단순 README가 아니라, 이후 작업에 다시 주입할 수 있는 구조화된 작업 컨텍스트다.

### 기능 B

명세서별 채팅으로 명세를 수정하고 작업으로 이어가는 기능도 구현 가능하다.

다만 아래 원칙이 필요하다.

- 채팅 로그와 명세 본문은 분리 저장
- 실행 기준은 항상 최신 승인 또는 확정 명세
- 채팅은 문서 수정 입력 채널이지 실행 기준 문서가 아님
- 채팅 결과는 diff나 patch 형태로 검토 후 반영

## 핵심 제품 흐름

가장 단순한 MVP 흐름은 아래다.

1. 프로그램 실행
2. 로컬 프로젝트 폴더 선택
3. 프로젝트 분석
4. 프로젝트 요약 생성
5. 명세 초안 생성
6. 명세서 채팅으로 수정
7. 수정안 diff 확인
8. 명세 반영 및 저장

초기 MVP에서는 여기까지 만들어도 충분히 가치가 있다.

## 현재 시점 추천안

- 플랫폼: `Electron`
- UI: `React + TypeScript`
- 저장 방식: 프로젝트 내부 `.sdd/`
- 명세 형식: markdown 본문 + 구조화 메타데이터
- 채팅 반영 방식: 제안 -> diff -> 반영 -> 버전 파일 저장
