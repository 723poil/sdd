# MVP Plan

## MVP 범위

첫 구현은 "코드를 자동 수정하는 앱"이 아니라 "로컬 프로젝트를 읽고 명세를 정리하는 앱"에 집중한다.

MVP의 끝점은 "CLI 에이전트가 실제 코드를 수정하는 것"이 아니라 "명세와 프로젝트 문맥을 정리해 다음 작업에 넘길 준비를 마치는 것"이다.

### 포함

- 로컬 UI 실행
- 앱 전역 CLI 에이전트 연결 설정
- 로컬 프로젝트 폴더 선택
- 연결된 CLI 에이전트를 통한 프로젝트 분석
- 프로젝트 개요 생성
- 명세 초안 자동 생성
- 명세서 목록/조회/저장
- 채팅
- 채팅 기반 수정안 생성
- diff 표시
- 명세 반영
- 로컬 저장 및 재진입

### 제외

- 다중 사용자 협업
- 클라우드 동기화
- 복잡한 권한 모델
- 고급 영향 분석
- Codex 실제 코드 수정 자동화
- 테스트 자동 실행
- Git commit 연동

## MVP 화면 흐름

### 0. 앱 설정 화면

- CLI 에이전트 연결 설정
- `Codex CLI`, `Claude Code`, `Gemini CLI` 연결 확인과 저장
- PATH 자동 감지 또는 직접 실행 경로 지정
- macOS에서는 PATH 외에 대표 설치 위치도 함께 탐색
- 인증 방식 기록
- 모델 확인 및 조정
- 추론 강도는 지원하는 에이전트에서만 확인 및 조정
- 연결 확인

주의:

- 이 화면은 실제 분석/채팅 실행과 분리한다
- 프로젝트 내부 `.sdd/`가 아니라 앱 전역 설정 파일을 사용한다

### 1. 시작 화면

- 최근 프로젝트 목록
- `프로젝트 폴더 선택`
- 빈 상태 안내

참고:

- 최근 프로젝트 목록은 프로젝트 내부 `.sdd/`가 아니라 앱 전역 설정 파일로 관리한다.

### 2. 프로젝트 선택 / 가져오기

- 로컬 폴더 선택
- 경로 검증
- 기본 정보 표시
- `전체 분석`
- `참조 분석`

### 3. 분석 진행 화면

- 프로젝트 문맥 수집
- 로컬 정적 참조 분석 실행
- 필요하면 CLI 에이전트 입력 생성
- 구조/계층/연결성 분석
- 문서 맵 연결선과 파일별 참조 관계 수집
- 분석 산출물 저장
- 진행 상태 표시
- 분석 문서는 Markdown 기준으로 저장하고 읽는다
- 파일 참조 관계는 TS/JS/Kotlin/PHP/Java 기준의 정적 분석으로 먼저 확보한다
- 파일 인덱스는 `api/settlement/payment` 같은 feature cluster와 `module / controller / command-service / query-service / command-handler / query-handler / repository / test` 같은 역할을 함께 분리해 보여준다
- 로컬 참조 분석의 file index는 스캔된 소스 파일 전체를 기본 재료로 삼고, UI에서 미리보기/펼치기로 가시성을 조절한다
- CLI 에이전트가 없거나 실패해도 정적 참조 분석 결과는 저장 가능해야 한다
- 프로젝트 경로가 읽기 전용이어도 참조 분석은 저장 없이 화면에 임시 표시할 수 있어야 한다
- 필요하면 Mermaid 로 구조도, 참조 관계, 계층도, 시퀀스, 유즈케이스를 함께 남긴다
- 분석은 activity가 이어지는 동안 계속 진행 가능해야 하며, 단순 고정 10분 wall timeout에 묶지 않는다
- 무응답이나 정체 상태만 실패로 본다
- 저장소가 크거나 역할이 분리되어 있으면 subagent 또는 delegated worker 활용을 유도하되, 사용자에게는 하나의 통합된 분석 결과만 보여준다

## 1차 개발 순서

### Step 1. 앱 껍데기

- Electron 기본 앱
- 기본 창/UI
- 프로젝트 폴더 선택

### Step 2. 분석 파이프라인

- 프로젝트 문맥 수집
- Codex exec 분석 요청 생성
- 분석 프롬프트를 XML 섹션 구조로 조립
- 구조화 결과 파싱
- `.sdd/analysis/*` 저장
- 긴 분석은 progress/activity 기반으로 추적하고, inactivity 상태일 때만 중단 처리한다
- 분석 프롬프트에는 큰 저장소에서 subagent 활용을 허용 또는 권장하는 문구를 포함한다

### Step 3. 명세 초안 생성

- 프로젝트 개요 명세
- 구조 명세
- 계층 명세
- 연결성 명세

### Step 4. 채팅

- 명세 컨텍스트별 대화 저장
- 세션은 명세 단위로 연결
- 새 명세는 별도 폼보다 `새 명세 채팅 시작`으로 초안을 연다
- 새 명세 생성 직후에는 번호가 붙은 버전을 만들지 않고 버전 없는 작업 초안으로 저장한다
- 채팅 응답은 user-facing reply 와 함께 최신 명세 초안을 구조화된 결과로 반환한다
- 사용자는 중앙 명세 문서에서 제목과 Markdown 본문을 직접 수정하고, 실질 변경이 있을 때만 새 버전으로 저장할 수 있어야 한다
- 첫 저장 또는 첫 채팅 반영이 `v1` 이고, 이후에도 변경 없음이면 새 버전을 만들지 않는다
- 명세 상세 화면은 버전 기록 조회, 현재 초안 대비 비교, 이전 버전 적용, 과거 버전 삭제를 지원한다
- 편집 화면은 작성 / 미리보기 / 분할 모드를 지원하고, 본문 복사는 renderer 로컬 동작으로 처리한다
- 명세 초안은 고정된 Markdown 형식을 따르고, 참조 태그 / 영향도 분석 / 사용 스킬 섹션을 포함해야 한다
- 영향도 분석은 참조 맵과 파일 인덱스를 근거로 관련 파일, 모듈, 흐름, 리스크를 정리해야 한다
- 사용 스킬 섹션은 `AGENTS.md` 와 `.codex/skills/` 를 참고해 이번 명세와 구현에 사용할 프로젝트 전용 스킬을 적는다
- 우측 채팅 입력 영역에서 현재 모델과 추론 강도를 바로 확인하고 조정
- 현재 선택된 에이전트/모델/추론 강도를 전송 인자에 반영해 해당 CLI 에이전트 exec 로 assistant 응답 생성
- user / assistant turn 을 세션 로그에 append-only 로 저장
- 수정안 생성
- diff 후 반영

### Step 5. 로컬 저장과 재진입

- `.sdd/` 생성
- 분석 결과 파일 저장
- 명세/채팅 재로드
- 내부 채팅 저장 구조는 세션형 로그를 재사용할 수 있지만, 사용자에게는 그냥 채팅으로 노출

### Step 6. CLI 에이전트 연결 준비

- 앱 전역 CLI 연결 설정 저장
- CLI 연결 확인
- Codex 는 `~/.codex/config.toml` 기본값 읽기
- 확정 명세 + 프로젝트 컨텍스트 조립
- 다음 작업용 입력 준비
- 선택된 CLI 에이전트 exec 기반 분석/명세/채팅 실행 경로 연결
- 장기 분석 실행 취소와 subprocess 종료 상태 반영

주의:

- 이 단계는 실제 코드 자동 수정이 아니라 실행 준비 단계다.
- 실제 코드 수정 자동화는 MVP 다음 단계로 둔다.
- 채팅 UI는 중앙 하단보다 우측 패널을 우선한다.

## 현재 추천 스택

- 플랫폼: `Electron`
- UI: `React + TypeScript`
- 저장: 프로젝트 내부 `.sdd/`
- 명세 저장: `markdown + json`
- 채팅 저장: `jsonl`
