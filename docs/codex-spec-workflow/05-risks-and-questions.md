# Risks And Questions

## 핵심 리스크

### 1. 분석 결과 정확도 부족

대응:

- confidence 표시
- unknowns 유지
- 추정과 확정을 구분

### 2. 채팅이 길어질수록 명세 품질 흔들림

대응:

- 채팅 로그와 명세 본문 분리
- patch/diff 중심 반영
- 명세 템플릿 유지

### 3. 명세와 실제 구현 준비 상태가 다시 어긋남

대응:

- 확정 명세 기준으로만 다음 단계 진행
- 후속 수정 필요 사항 기록
- 결과 요약을 남김

### 4. 프로젝트 분석 비용 증가

대응:

- 초기 전체 분석 + 이후 증분 분석
- 캐시 분리
- 관련 파일 후보 먼저 계산

### 5. 파일 기반 저장 구조가 복잡해짐

대응:

- `.sdd/` 구조 고정
- `meta.json + versions/ + chat.jsonl` 규칙 유지
- 사람이 읽기 쉬운 파일명 사용

## 구현 전에 확정할 결정

### A. 플랫폼

- `Electron`
- `Tauri`

현재 기준 추천은 `Electron`이다.

### B. 분석 기준 스냅샷

- 로컬 파일 시스템 현재 상태
- Git commit 기준
- 수동 snapshot

초기에는 "마지막 분석 시점의 로컬 상태"를 snapshot으로 간주하고 시작해도 된다.

### C. 명세 최소 구조

권장 섹션:

- 목표
- 배경
- 범위
- 사용자 시나리오
- 기능 요구사항
- acceptance criteria
- 영향 받는 영역
- 오픈 이슈

### D. 반영 방식

권장:

- 채팅 -> 수정안 생성
- diff 확인
- 반영
- 새 버전 저장

### E. Codex 자동화 범위

후속 단계에서 아래 중 어디까지 갈지 정해야 한다.

- 명세 초안 작성만
- 구현 계획까지
- 실제 코드 수정까지
- 테스트 결과 정리까지

## 문답식 보완 질문

### Q1. Electron으로 시작할지, Tauri로 시작할지

현재 추천:

- Electron

이유:

- 로컬 파일 접근과 앱 내부 로직 구성이 단순함

### Q2. `.sdd/`를 git에 포함할지 제외할지

선택지:

- 항상 포함
- 기본 제외
- 파일별로 구분

이 결정은 협업 방식과 이력 관리 방식에 영향을 준다.

현재 상태:

- 아직 미정
- 문서상 기본값으로 고정하지 않고 열어둔다

### Q3. 명세서 종류를 몇 개로 시작할지

권장 초기 세트:

- 프로젝트 개요 명세
- 구조 명세
- 기능 작업 명세

### Q4. 채팅 반영은 항상 수동 승인인지

권장:

- 항상 수동 승인

### Q5. 첫 MVP에서 Codex 실제 코드 수정을 포함할지

권장:

- 포함하지 않음
- 우선은 작업 준비용 명세 도구에 집중

## 다음 문답 진행 방식

앞으로는 아래 방식으로 문서를 계속 고도화하면 된다.

1. 질문 1개 또는 2개 확정
2. 관련 문서에 바로 반영
3. 구조와 규칙을 한 단계 더 구체화

현재 가장 먼저 확정하면 좋은 것은 아래다.

1. `Electron` 확정 여부
2. `.sdd/`의 git 포함 정책
3. Codex 자동화 범위
